Tally Invoice Tracker

Full-stack project walkthrough · Architecture · Q&A · Key numbers

Vue 3 · ASP.NET Core 9 · PostgreSQL · Redis · Docker · AWS · Terraform

1. The One-Sentence Pitch

"Tally is a full-stack invoice management SaaS — users create, send, and track invoices for their clients, and the system automatically marks unpaid invoices overdue each night."

Use this as your opening line every time. It tells the panel: what it does, who uses it, and that there's a non-trivial background process. Everything else is detail you layer on top.

2. Panel Walkthrough Script

Tell this as a flowing story, not a bullet list. Aim for about 3 minutes.

Opening

"The application has three main layers — a Vue 3 SPA on the frontend, an ASP.NET Core 9 REST API as the backend, and PostgreSQL as the database. Redis is also in the stack for token revocation. Everything is containerised with Docker Compose and deployed on an AWS EC2 instance via Terraform."

Authentication

"A user registers and receives a short-lived JWT access token — 25 minutes — plus a long-lived refresh token — 7 days. The frontend silently refreshes the access token 60 seconds before it expires, so the session feels seamless. When they log out, the current JWT's unique ID — the jti claim — is placed on a Redis blacklist with a TTL equal to the token's remaining lifetime, so it can't be replayed even before it expires."

Backend Architecture

"Inside the API I followed a layered architecture: Controllers handle HTTP concerns, Services hold business logic, and Repositories handle data access — all wired up via .NET's built-in dependency injection. Every layer talks through interfaces, which makes unit testing straightforward without spinning up a real database."

Interesting Design Problem

"One interesting design problem was invoice number generation. If two requests come in simultaneously you can't let them both read the same counter and increment separately — you'd get duplicate invoice numbers. I solved it with a PostgreSQL INSERT ... ON CONFLICT DO UPDATE ... RETURNING — a single atomic upsert that increments and returns the new counter in one round trip, with no race condition and no retry logic needed."

State Machine

"Invoice status follows a state machine — a Draft can be Sent or Cancelled, a Sent invoice can go to Paid, Overdue, or Cancelled, but you can never go backwards. A background service runs nightly at midnight UTC and bulk-updates any Sent invoices past their due date to Overdue."

Infrastructure

"On the infrastructure side, Terraform provisions an EC2 instance, attaches an Elastic IP for a stable DNS target, and creates an AWS Secrets Manager secret. The EC2 user-data script installs Docker and writes a fetch-secrets script that pulls credentials at deploy time — so secrets never live in the repo or in Terraform state. Nginx sits in front of everything — it terminates TLS with a Let's Encrypt certificate, serves the Vue static files, and reverse-proxies /api/ requests to the backend container. Because both the UI and API are on the same origin, there's no CORS configuration needed at all."

4. Architecture Patterns

Layered / Clean Architecture

Controller → Service → Repository → Database. Each layer only talks to the one below it via an interface. Swapping the database or faking a repository in tests requires no changes to the service layer.

Repository Pattern

IInvoiceRepository, IUserRepository, IClientRepository. Services never touch DbContext directly. Benefit: tests inject fakes without a real DB. Trade-off: extra boilerplate on top of EF Core, which is already a repository internally.

DTO + AutoMapper

Domain models (Invoice, User) never cross the HTTP boundary. DTOs are the API contract. AutoMapper converts between them. Changing an internal field name doesn't break the API response shape.

State Machine (Invoice Status)

The AllowedTransitions dictionary defines which status changes are legal. Any invalid transition throws InvalidOperationException → controller catches it → returns HTTP 422. Terminal states (Paid, Cancelled) have empty allowed sets.

Draft     → [Sent, Cancelled]
Sent      → [Paid, Overdue, Cancelled]
Overdue   → [Paid, Cancelled]
Paid      → []          ← terminal
Cancelled → []          ← terminal

Background Service (Hosted Service)

OverdueInvoiceJob extends BackgroundService. It waits until midnight UTC, then issues a single bulk EF Core ExecuteUpdateAsync — one SQL UPDATE for all affected rows, no per-invoice round trip. No external scheduler (Hangfire, Quartz) needed.

Token Revocation via Redis Blacklist

JWTs are stateless — you can't "delete" them server-side. On logout, the JWT's jti claim is written to Redis with a TTL equal to the token's remaining lifetime. OnTokenValidated checks Redis on every request. When the TTL expires, Redis removes the key automatically.

Atomic Counter via PostgreSQL Upsert

No SELECT + UPDATE two-step (race condition). One atomic INSERT ... ON CONFLICT DO UPDATE ... RETURNING. PostgreSQL serialises the conflict resolution — even under concurrent requests, no duplicates, no gaps, no retry logic.

INSERT INTO "InvoiceCounters" ("UserId", "YearMonth", "Counter")
VALUES ($1, $2, 1)
ON CONFLICT ("UserId", "YearMonth") DO UPDATE
    SET "Counter" = "InvoiceCounters"."Counter" + 1
RETURNING "Counter"

Single-Origin Deployment (no CORS)

Nginx serves the Vue SPA at / and proxies /api/* to the backend — both from the same domain (tally.inkcat.net). The browser never makes a cross-origin request, so CORS headers are completely unnecessary and intentionally removed.

8. Interview Q&A

1Walk me through the authentication system.

Authentication uses a two-token pattern. The user logs in and gets a short-lived JWT access token — 25 minutes — and a long-lived refresh token — 7 days. The access token is a signed JWT containing the user's ID, email, role, and a unique jti claim. The refresh token is a cryptographically random 64-byte value — only its SHA-256 hash is stored in the database, so a database breach doesn't expose the tokens themselves.

The frontend stores both in localStorage and schedules a silent refresh 60 seconds before the access token expires. When the user logs out, two things happen: the current JWT's jti is written to Redis with a TTL equal to its remaining lifetime, and the refresh token is revoked in the database. Every authenticated request's OnTokenValidated event checks Redis, so a token can't be replayed after logout even if it hasn't technically expired yet.

2Why did you use the Repository pattern? Isn't it redundant over EF Core?

That's a fair critique — EF Core's DbContext is already a Unit of Work and the DbSet is already a repository. The reason I added it anyway was testability. By coding against IInvoiceRepository instead of AppDbContext, the service tests can inject a simple fake or in-memory implementation without spinning up a real database.

In the test project you can see FakeInvoiceNumberAllocator — same principle. The trade-off is a bit more boilerplate, but for a project with real test coverage I think it's worth it. If I were building a much simpler CRUD app with no tests, I'd probably skip the repositories and use DbContext directly in the services.

3How does invoice number generation work and why did you design it that way?

Invoice numbers follow a per-user monthly format: INV-202605-0001. The naive approach — read the counter, increment it, write it back — creates a race condition. Two simultaneous requests could both read the same counter value and produce duplicate numbers.

Instead I use a single PostgreSQL INSERT ... ON CONFLICT DO UPDATE ... RETURNING. If no counter row exists for that user and month it inserts one with Counter = 1. If one already exists it atomically increments it. The RETURNING clause gives me the new value in the same round trip. Because PostgreSQL serialises the conflict resolution internally, there's no race condition and no retry logic needed — it just works under any level of concurrent load.

4Explain the invoice status state machine.

An invoice has five states: Draft, Sent, Overdue, Paid, and Cancelled. Not all transitions are valid. I encoded the allowed transitions as a dictionary in InvoiceService: Draft → [Sent, Cancelled], Sent → [Paid, Overdue, Cancelled], Overdue → [Paid, Cancelled], and the terminal states Paid and Cancelled have empty allowed sets.

If you attempt an invalid transition the service throws an InvalidOperationException which the controller catches and returns as HTTP 422 Unprocessable Entity with a human-readable message. The nightly background job uses ExecuteUpdateAsync to bulk-transition any Sent invoices past their due date directly in the database in a single SQL UPDATE — no round-trip per row.

5How is the app secured?

Security is layered. At the network level nginx enforces TLS 1.2+, sets HSTS, and rate-limits the auth endpoints to 5 requests per minute per IP to slow brute-force attacks. The API endpoints are protected with JWT bearer auth and role-based authorization — admin endpoints require the Admin role.

All user inputs go through a Sanitizer class that strips HTML tags using a compiled regex before persisting — defence against stored XSS. Passwords are BCrypt hashed. Reset tokens and refresh tokens are stored as SHA-256 hashes. The API enforces a 1 MB request body limit to prevent large-payload DoS. The global exception handler returns a generic 500 message — no stack traces ever reach the client. In Docker the API runs as a non-root user. The AWS IAM role grants only secretsmanager:GetSecretValue on the one specific secret — least privilege throughout.

6Why Vue 3 + Pinia instead of React + Redux?

Vue 3's Composition API feels very natural for this kind of CRUD application. ref and computed map cleanly onto reactive data and derived values. Pinia is the official Vue 3 state management solution — it's much simpler than Vuex, fully TypeScript-typed out of the box, and stores are just functions that return reactive state and actions. There's no boilerplate mutations layer.

The honest trade-off is that React has a larger ecosystem and more job-market demand. But for a self-contained project Vue gave me a faster, more pleasant development experience, and the Composition API composables are just as testable as React hooks.

7What is the purpose of Redis in this project?

Redis serves two purposes. First, it's the token revocation store — when a user logs out, the JWT's jti is stored as a Redis key with a TTL equal to the token's remaining lifetime. When the TTL expires Redis automatically deletes the key, so there's no cleanup job needed.

Second, the output cache for the summary endpoint uses in-memory caching — the summary aggregates across all invoices and is expensive to recompute on every request. It's cached for 30 seconds per user, with the cache key varied by the Authorization header so users never see each other's data.

8Explain your Terraform and deployment approach.

Terraform provisions the AWS infrastructure: an EC2 t3.small running Ubuntu 24.04, a security group allowing ports 22/80/443, an Elastic IP for a stable DNS A-record, and an IAM role attached to the instance with permission only to call GetSecretValue on a specific Secrets Manager secret.

The EC2 user-data script installs Docker on first boot and writes a fetch-secrets.sh script. On every deploy, that script calls the Secrets Manager API, gets the JSON secret, and uses jq to write it as a .env file for Docker Compose to read. Credentials are never in the repository, never in Terraform state — they live only in Secrets Manager and briefly in memory during deploy.

9How does the frontend handle token expiry gracefully?

When the frontend receives a login or refresh response, it decodes the JWT payload — the middle Base64 segment — to read the exp claim and stores the expiry timestamp in localStorage. It then schedules a setTimeout to fire 60 seconds before expiry. That timer calls a silent refresh — it hits /api/auth/refresh with the stored refresh token, gets a new access token back, updates the store and localStorage, and schedules the next refresh cycle.

The user never sees a logout or authentication error unless the refresh token itself has expired or been revoked, in which case the store clears the session and redirects to /login.

10What would you improve or do differently?

HttpOnly cookies for refresh tokens. Currently the refresh token is stored in localStorage, which is accessible to JavaScript and therefore vulnerable to XSS. A better approach would be an HttpOnly cookie — JavaScript can't read it at all. There's actually a TODO comment in AuthService for this.

Password reset email. The reset token is generated and stored, but the email sending isn't wired up yet — another TODO in the code.

Migration job, not startup migration. Running Database.Migrate() on startup can cause a race condition in a multi-instance deployment. A separate pre-deploy migration job would be safer.

Async messaging. There's a TODO for RabbitMQ for async invoice processing. At high volume you'd want invoice creation to publish an event rather than doing everything synchronously in the request pipeline.

10. Tips for the Day

Draw the diagram early. If there's a whiteboard, sketch the 4-container stack (nginx, api, postgres, redis) in the first minute. It gives the panel a shared mental model and makes all your other answers easier to follow.
Name the trade-offs. SE3 panels want to see engineering judgment, not just "I used X because it's good." For every choice, have one sentence about what you gave up — e.g., "Pinia is simpler but React has a larger ecosystem."
The TODO comments are honest and good. HttpOnly cookies for refresh tokens, the unwired email reset, the startup migration race — bring these up yourself in Q10. It shows self-awareness and that you understand the gap between a working prototype and production-ready code.
On the atomic counter: The key insight is "one round trip, no retry, no gap." Practice saying those three things smoothly — they signal you actually understand why the naive approach fails under concurrency.
On Redis: The TTL auto-expiry is the elegant part — "when the token would have expired anyway, the revocation key disappears too — zero cleanup work." That detail impresses.
On CORS: Most candidates add CORS without thinking. Being able to explain why you deliberately removed it ("same-origin because nginx proxies both under one domain") shows architectural thinking.
