1. Microservices — Pros & Cons

A microservices architecture decomposes an application into a set of small, independently deployable services, each owning its own data store and communicating over a network (HTTP/REST, gRPC, or message queues).

How it works

Each service is a separate process — potentially written in a different language, deployed on different hardware, scaled independently. A service mesh or API gateway often sits in front to handle routing, auth, and observability. Services discover each other via DNS, a registry (Consul, Eureka), or a cloud load balancer.

Pros

  • Independent deployability — change and ship the billing service without touching the user service
  • Independent scalability — scale only the bottleneck (e.g. PDF rendering) without scaling the whole app
  • Technology heterogeneity — use Python for ML, Go for high-throughput ingestion, C# for the business logic
  • Fault isolation — a crash in one service doesn't bring down the entire system (with circuit breakers)
  • Team autonomy — teams own their service end-to-end; Conway's Law becomes a feature
  • Smaller codebases — each service is easier to understand, test, and onboard into

Cons

  • Distributed systems complexity — network calls fail; you need retries, timeouts, circuit breakers, idempotency
  • Data consistency — no cross-service transactions; you use eventual consistency and Saga patterns instead
  • Operational overhead — each service needs its own CI/CD pipeline, health checks, logging, tracing, and deployment config
  • Latency — an in-process function call (nanoseconds) becomes an HTTP call (milliseconds)
  • Harder to debug — a request spans multiple services; you need distributed tracing (Jaeger, Zipkin) to follow it
  • Over-engineering for small teams — the overhead rarely pays off below ~50 engineers
In Tally's context: Tally is a monolith — the right call for a solo-built project. If it grew, you'd first extract the PDF generation service (CPU-intensive, independent) and the email service (async, no strong consistency requirements) before splitting anything else.

2. Monolith — Pros & Cons

A monolith deploys as a single unit. All modules share a process, memory space, and database. "Modular monolith" is a middle ground — strong internal boundaries (namespaces, project layers) but still a single deployable.

Pros

  • Simple to develop — one repo, one debugger, one deploy pipeline, no network calls between modules
  • ACID transactions — you can wrap multiple business operations in a single DB transaction with no saga choreography
  • Easy refactoring — rename a class or change an interface; the compiler tells you everywhere to fix
  • Low latency — function calls, not network hops
  • Straightforward testing — integration tests spin up the whole app in-process
  • Lower infrastructure cost — one container/process to run, monitor, and log

Cons

  • Scaling is all-or-nothing — you scale the entire app even if only one module is under load
  • Long build/test cycles — as the codebase grows, CI slows down because everything must be rebuilt
  • Deployment risk — every deploy touches the whole system; one bad change affects everything
  • Technology lock-in — if you chose C# you're using C# everywhere
  • Team coupling — large teams step on each other; merge conflicts are common
  • Big bang failure — a crash or OOM kills the whole application
Modular monolith pattern: Enforce strong module boundaries (separate projects/namespaces, interfaces-only cross-module communication, no direct DB access across modules) so that if you ever do split, the seams are already there. Tally's layered architecture (Controllers → Services → Repositories) is a step in this direction.

3. JS Call Stack Budget

The call stack is a LIFO (last-in, first-out) data structure that the JavaScript engine uses to track where in the program execution currently is. Every function call pushes a stack frame onto the stack; every return pops one off.

The budget

Every environment (V8, SpiderMonkey, JavaScriptCore) imposes a fixed maximum stack depth — typically around 10,000–15,000 frames in Node.js. Exceeding it throws a RangeError: Maximum call stack size exceeded.

Why it matters

Each frame stores local variables, the return address, and arguments. Stack memory is a limited, contiguous region — unlike heap memory it can't grow dynamically. The engine pre-allocates it for predictable performance.

// Stack overflow — unbounded recursion
function infinite() {
  return infinite(); // each call pushes a frame, never pops
}
infinite(); // RangeError after ~10k frames

// Fix 1: trampolining — return a function instead of calling it
function trampoline(fn) {
  let result = fn;
  while (typeof result === 'function') result = result();
  return result;
}

// Fix 2: tail-call optimisation (TCO)
// ES6 spec defines TCO but V8 does NOT implement it in practice.
// Safari/JavaScriptCore does. Don't rely on it in Node.js.

// Fix 3: iterative approach — convert recursion to a loop
function factorial(n) {
  let acc = 1;
  for (let i = 2; i <= n; i++) acc *= i;
  return acc;
}
Interview gotcha: TCO is in the ES6 spec but V8 intentionally does not implement it (it makes stack traces impossible to read). Always assume Node.js does NOT have TCO.

4. The Event Loop

JavaScript is single-threaded. The event loop is the mechanism that lets a single thread handle thousands of concurrent I/O operations without blocking.

Architecture

┌─────────────────────────────────────────────────────────┐ │ Call Stack │ │ (synchronous code executes here, one frame at a time) │ └───────────────────────────┬─────────────────────────────┘ │ stack empty? ▼ ┌───────────────────────────────────────────────────────────┐ │ Event Loop │ │ │ │ 1. microtask queue (Promise callbacks, queueMicrotask) │ ← drained completely first │ 2. macrotask queue (setTimeout, setInterval, I/O cb) │ ← one task per loop tick │ 3. check phase (setImmediate — Node.js only) │ │ 4. close callbacks (socket.on('close', ...)) │ └───────────────────────────────────────────────────────────┘

The phases in detail

  • timers — executes setTimeout and setInterval callbacks whose threshold has elapsed
  • pending callbacks — I/O errors from the previous loop iteration
  • idle/prepare — internal Node.js use only
  • poll — retrieves new I/O events; if none are ready, blocks here waiting
  • check — setImmediate callbacks run here (always after the current poll phase)
  • close callbacks — e.g. socket.destroy()

Microtask vs macrotask

After every macrotask, the event loop drains the entire microtask queue before moving on. This means chained .then() callbacks all run before the next setTimeout fires, even a setTimeout(fn, 0).

console.log('1');
setTimeout(() => console.log('4 - macrotask'), 0);
Promise.resolve().then(() => console.log('2 - microtask'));
queueMicrotask(() => console.log('3 - microtask'));
// Output: 1, 2, 3, 4
Blocking the event loop: Any synchronous operation that takes >1 ms starves all pending I/O. Never run CPU-heavy work (crypto, image processing, JSON parsing of huge payloads) on the main thread in Node.js — offload to a Worker Thread or a child process.

5. Node.js Concurrency

Node.js achieves high concurrency with a single thread by delegating I/O to the OS (via libuv) and processing results asynchronously on the event loop. It does NOT run JavaScript in parallel — but it doesn't need to for I/O-bound work.

libuv and the thread pool

When you call fs.readFile(), Node doesn't wait — it hands the syscall to libuv, which uses OS async I/O (epoll on Linux, kqueue on macOS, IOCP on Windows) or a background thread pool for operations that don't have native async syscalls (DNS lookup, some crypto, file system). The default thread pool size is 4 threads (configurable via UV_THREADPOOL_SIZE, max 1024).

User code (JS thread) │ ├── fs.readFile() ──────────────▶ libuv (OS async or thread pool) ├── https.get() ──────────────▶ libuv (OS async networking) ├── bcrypt.hash() ─────────────▶ libuv thread pool │ Event loop polls libuv for completed work │ └── fires callback when I/O is done

Worker Threads (Node 12+)

For true CPU parallelism in Node.js use worker_threads. Workers run in separate V8 isolates with their own event loop, communicating via postMessage. Useful for: image processing, parsing, machine learning inference, any CPU-bound task.

Cluster module

The cluster module forks N worker processes (each with their own event loop and V8 heap), sharing the same port. The master distributes incoming connections. PM2 abstracts this. This is how Node.js scales across CPU cores on a single machine — multiple processes, not threads.

Why this is irrelevant to Tally's backend: Tally uses ASP.NET Core, which is truly multi-threaded and handles concurrency natively via the CLR thread pool. Each HTTP request runs on a pool thread. async/await in C# releases the thread back to the pool while awaiting I/O, allowing tens of thousands of concurrent requests with a small thread pool.

6. Pausing in a Promise

A Promise is an object representing the eventual result of an async operation. It is in one of three states: pending, fulfilled, or rejected. Once settled it is immutable.

How await pauses

await is syntactic sugar over .then(). When the JS engine hits await expr:

  1. It evaluates expr to get a Promise (or wraps a non-Promise in Promise.resolve()).
  2. It suspends the current async function — saving its local state onto the heap (not the call stack).
  3. It returns a new Promise to the caller immediately, freeing the call stack.
  4. When the awaited Promise settles, a microtask is queued to resume the suspended function.
async function fetchData() {
  console.log('A');           // runs synchronously
  const data = await fetch('/api/data'); // suspends HERE — returns to event loop
  console.log('B');           // runs after fetch resolves (microtask)
  return data.json();
}

fetchData();
console.log('C'); // runs before B — fetchData() suspended, caller continues
// Output: A, C, B

Pitfalls

  • Sequential awaits when parallel is possible: await a; await b; runs serially. Use Promise.all([a, b]) to run concurrently.
  • Unhandled rejections: a rejected Promise with no .catch() or try/catch around await crashes Node.js in recent versions.
  • await in loops: for...of with await is sequential. Use Promise.all(arr.map(async item => ...)) for parallel iteration.
// ❌ Sequential — waits for each one to finish before starting next
for (const id of ids) {
  const result = await fetchInvoice(id);
}

// ✅ Parallel — all requests in flight simultaneously
const results = await Promise.all(ids.map(id => fetchInvoice(id)));

ASP.NET Core's model

ASP.NET Core uses the CLR thread pool. Each request is dispatched to a pool thread. When an async handler hits an await, it releases its thread back to the pool (thread is free to serve another request). When the I/O completes, any available thread resumes the continuation. This gives you the throughput of async I/O without Node's event loop constraint — CPU-bound work doesn't block other requests because the CLR scheduler will context-switch threads.

// C# — awaiting releases the thread
[HttpGet("{id}")]
public async Task<IActionResult> GetInvoice(int id)
{
    // Thread returns to pool while EF Core waits for Postgres
    var invoice = await _invoiceRepo.GetByIdAsync(id);
    return Ok(invoice);
}

8. Nginx — Pros, Cons & Alternatives

Nginx (engine-x) is an event-driven, asynchronous web server and reverse proxy. It uses a master process + worker processes model where each worker handles thousands of connections in a single thread using epoll/kqueue.

In Tally

Nginx serves three roles: (1) terminates TLS with a Let's Encrypt certificate, (2) serves the compiled Vue SPA from the filesystem, (3) reverse-proxies /api/ requests to the ASP.NET Core container on port 8080. Because both UI and API are on the same origin, there is no CORS configuration anywhere in the stack.

Pros

  • High concurrency at low memory — event-driven; handles 10k+ connections per worker
  • Static file serving — extremely fast; serves files directly from OS page cache
  • TLS termination — offloads SSL handshakes from the application
  • Reverse proxy / load balancer — upstream blocks, health checks, weighted round-robin
  • Request buffering — buffers slow client uploads; app server only sees complete requests
  • Gzip/Brotli compression — transparent to the app
  • Mature and battle-tested — ~34% of all web servers worldwide

Cons

  • Config syntax is not intuitive — location block matching rules have surprising precedence
  • Modules are compile-time — unlike Apache, you can't add modules at runtime
  • Lua scripting required for complex logic (OpenResty variant)
  • Limited dynamic routing — not a full API gateway; no built-in rate limiting per user (requires Lua or Plus)
  • No built-in service discovery — upstream IPs must be known at config time (or you use a resolver)

Alternatives

9. ORM — Pros & Cons

An ORM (Object-Relational Mapper) maps between object-oriented class hierarchies and relational database tables, handling SQL generation, connection management, and result mapping.

In Tally: EF Core with the Npgsql provider. Entities are C# classes; migrations track schema changes; LINQ queries are translated to SQL at runtime.

Pros

  • Type safety — LINQ queries are checked by the compiler; no SQL strings to mistype
  • Code-first migrations — schema lives in source control alongside the code; dotnet ef migrations add generates SQL diffs
  • Productivity — CRUD operations are one-liners; no boilerplate result mapping
  • Change tracking — EF Core tracks entity state; calling SaveChangesAsync() batches all changes
  • Database portability — swap Npgsql for SQLite for in-memory testing with minimal code changes
  • Relationships — navigation properties; Include() for eager loading

Cons

  • N+1 queries — forgetting .Include() and lazy-loading inside a loop issues one query per row
  • Generated SQL can be inefficient — complex LINQ sometimes generates suboptimal SQL; you need to profile
  • Leaky abstractions — you still need to understand SQL to write performant queries and read migration output
  • Overhead for bulk operations — updating 10,000 rows via EF Core is slower than a single UPDATE WHERE; use ExecuteUpdateAsync() or raw SQL
  • Magic behaviour — change tracking, lazy loading, and cascade deletes can cause surprising side effects
In Tally's OverdueInvoiceJob: The background job uses ExecuteUpdateAsync() — a bulk update that bypasses change tracking and issues a single SQL UPDATE statement. This is intentional: loading all overdue invoices into memory to set their status one-by-one would be an N+1 anti-pattern.

10. JVM vs Compiler (AOT)

The JVM model (bytecode + JIT)

Java and Kotlin source code compiles to bytecode — a platform-neutral instruction set. The JVM interprets this bytecode and, over time, the JIT (Just-In-Time) compiler identifies hot paths and compiles them to native machine code. The JVM also manages garbage collection.

  • Write once, run anywhere — the same .jar runs on any OS with a compatible JVM
  • Warm-up cost — the JIT needs time to profile and compile; cold-start is slow (important for serverless/containers)
  • Peak performance can be excellent — HotSpot JIT can surpass AOT for long-running processes by exploiting runtime profiling data (e.g. inlining virtual calls based on observed call sites)

AOT compilation (native compiler)

C, C++, Rust, and Go compile directly to native machine code for a specific target architecture. There is no runtime interpreter. The binary is self-contained.

  • Instant startup — no JIT warm-up; good for CLIs, FaaS, containers
  • Predictable performance — no GC pauses (in manual-memory or arena-allocation languages)
  • Platform-specific — you compile separately for each target

Where does .NET sit?

.NET uses both. The CLR (Common Language Runtime) has a JIT compiler (RyuJIT) — the same bytecode (IL) runs on any platform, JIT-compiled at runtime. Since .NET 6, Native AOT is available — you can compile a .NET app directly to a native binary with no CLR runtime, achieving faster startup and smaller containers. Tally uses the standard JIT path (net9.0).

11. ASP.NET Core — Why .NET 9, Not 8 or 10

Why ASP.NET Core at all

  • Strongly typed end-to-end (C# → EF Core → PostgreSQL)
  • Excellent built-in DI container; middleware pipeline is composable and explicit
  • First-class async/await with the CLR thread pool — handles I/O-bound CRUD extremely efficiently
  • Rich ecosystem: EF Core, JWT Bearer, Serilog, AutoMapper, MailKit all integrate cleanly
  • Health checks, output caching, OpenAPI support built into the framework

Why .NET 9 over .NET 8 for Tally: EF Core 9 introduces ExecuteUpdateAsync refinements, complex type mapping improvements, and the built-in OpenAPI document generation removes the Swashbuckle dependency. .NET 10 is the next LTS release (November 2025, supported to 2028). At the time Tally was built, .NET 10 was in preview — not suitable for a production project. When .NET 10 releases as LTS, upgrading from .NET 9 is straightforward (change the TargetFramework in the .csproj; breaking changes between STS and the next LTS are minimal by design).

12. PostgreSQL — Pros & Cons

PostgreSQL is a fully ACID-compliant, open-source object-relational database. It uses an MVCC (Multi-Version Concurrency Control) model — readers never block writers and writers never block readers by keeping multiple row versions.

Pros

  • Full ACID compliance — transactions are durable; no half-written rows
  • MVCC — high read-write concurrency without locking
  • Rich data types — JSONB, arrays, ranges, composite types, PostGIS for geospatial
  • Atomic upsert — INSERT ... ON CONFLICT DO UPDATE ... RETURNING — used in Tally's invoice counter
  • Robust indexing — B-tree, GIN (full-text, JSONB), GiST, partial indexes, expression indexes
  • Foreign keys & constraints — enforced at DB level, not just the ORM
  • Window functions, CTEs, lateral joins — powerful analytics without application-level aggregation
  • Strong community, excellent documentation

Cons

  • Write amplification — MVCC creates dead tuples; requires VACUUM (usually autovacuum) to reclaim space
  • Not natively distributed — scaling writes requires Citus, Patroni/Patroni, or logical replication; horizontal sharding is complex
  • Heavier than SQLite — requires a running server process; overkill for embedded or mobile use
  • Schema migrations require care — adding a NOT NULL column to a large table locks it (use ADD COLUMN DEFAULT in Postgres 11+ which is instant)
  • No built-in column-store — for OLAP workloads, TimescaleDB or an external DW (Redshift, BigQuery) is better
Tally's specific use of Postgres: The InvoiceCounters table uses an atomic INSERT ... ON CONFLICT DO UPDATE ... RETURNING to generate sequential invoice numbers without a distributed lock or retry loop. This is only possible in Postgres (and MySQL 8+). SQLite's locking model would serialize all inserts; a pure application-level counter would have a race condition.

13. Redis — Pros, Cons & Alternatives

Redis (Remote Dictionary Server) is an in-memory data structure store. It supports strings, hashes, lists, sets, sorted sets, streams, and more. Persistence is optional (RDB snapshots, AOF append-only log).

In Tally: Redis serves two purposes — JWT revocation blacklist (a SET of blacklisted JTI claims with TTL auto-expiry) and output caching of the /api/invoices/summary endpoint (30-second TTL).

Pros

  • Sub-millisecond latency — all data in RAM; no disk I/O on reads
  • TTL on every key — perfect for tokens, sessions, cache entries; expiry is automatic
  • Atomic operations — INCR, SETNX, LPUSH etc. are atomic; useful for distributed locks (Redlock) and counters
  • Pub/Sub and Streams — basic message queue without a separate broker
  • Cluster mode — horizontal scaling via consistent hashing
  • Mature ecosystem — StackExchange.Redis (.NET), ioredis (Node), Jedis (Java)

Cons

  • Data is in RAM — expensive for large datasets; not a primary store for business data
  • Single-threaded command processing — one slow Lua script blocks all other commands
  • Persistence tradeoffs — RDB = potential data loss on crash; AOF = slower writes
  • No complex queries — no JOINs, no LIKE searches (unless using RediSearch module)
  • Cluster complexity — multi-key operations must hit the same slot; requires careful key design

Alternatives

ToolUse caseDifference from Redis
MemcachedSimple key-value cacheFaster for pure caching (no persistence overhead); no data structures beyond strings; no pub/sub
DragonflyRedis-compatible drop-inMulti-threaded; claims 25× throughput; same Redis protocol
KeyDBRedis fork, multi-threadedOpen-source multi-threaded Redis fork; compatible API
Postgres (LISTEN/NOTIFY)Lightweight pub/subIf you already have Postgres, LISTEN/NOTIFY avoids adding Redis for simple event notifications

14. Caching — Strategies & Patterns

Caching stores computed or fetched results so future requests can be served without re-computing or re-fetching. The goal is to reduce latency and reduce load on slower backing stores (database, external APIs).

Cache-aside (lazy loading)

The application checks the cache first. On miss, it queries the DB, writes the result to the cache, then returns it. This is what Tally's summary endpoint does.

// Pseudocode
var cached = await redis.GetAsync(key);
if (cached != null) return cached;
var result = await db.ComputeSummaryAsync(userId);
await redis.SetAsync(key, result, ttl: 30s);
return result;

Write-through

On every DB write, the application also updates the cache. Cache is always fresh. Higher write latency; cache always has data even before first read.

Write-behind (write-back)

Writes go to cache immediately; the cache asynchronously flushes to the DB. Very low write latency but risk of data loss on crash.

Read-through

Cache sits in front of the DB. The cache itself handles the DB read on a miss — the application only ever talks to the cache. Common in managed services (ElastiCache DAX for DynamoDB).

Cache invalidation strategies

  • TTL expiry — stale data serves for up to TTL seconds; simplest, eventual consistency
  • Explicit invalidation — on write, delete or update the cache key; requires careful key design
  • Tag-based invalidation — ASP.NET Core's output cache tags; Tally invalidates the "summary" tag whenever an invoice is created, updated, or deleted
  • Cache-busting — versioned keys (summary:v3); deploy a new version and old data is ignored
"There are only two hard things in Computer Science: cache invalidation and naming things." — Phil Karlton. Stale cache serving incorrect data is often worse than no cache. Tag invalidation is safer than TTL-only for mutable business data.
In Tally: GET /api/invoices/summary uses ASP.NET Core's output caching with a SummaryCache policy (30s TTL, varies by Authorization header so each user gets their own cache entry). The cache is tagged "summary". The InvoicesController calls cache.EvictByTagAsync("summary") on any mutating operation, ensuring the summary is never stale after a write.

15. BCrypt — Pros, Cons & Alternatives

BCrypt is a password-hashing function designed in 1999 by Niels Provos and David Mazières. It is deliberately slow — it incorporates a cost factor (work factor) that can be increased as hardware gets faster. It also generates a random salt automatically and embeds it in the output hash.

How it works

BCrypt outputs a 60-character string: $2a$<cost>$<22-char-salt><31-char-hash>. The cost factor (11 in Tally's seed data: $2a$11$...) means 2¹¹ = 2,048 iterations of the Blowfish cipher. On modern hardware, a cost-11 hash takes ~100ms. Doubling the cost to 12 doubles the time to ~200ms.

Pros

  • Adaptive cost factor — increase cost as hardware improves without changing the algorithm
  • Built-in salt — each hash has a unique random salt; identical passwords produce different hashes
  • Resists GPU/ASIC cracking — memory-intensive enough to make GPU parallelisation less effective than for MD5/SHA
  • Widely audited — 25+ years of cryptanalysis; no practical attacks
  • Simple API — BCrypt.HashPassword() / BCrypt.Verify()

Cons

  • 72-byte input limit — BCrypt truncates passwords beyond 72 bytes; very long passwords are silently weaker
  • Not the newest recommendation — OWASP now recommends Argon2id as first choice
  • CPU-bound only — Argon2id is also memory-hard, making GPU cracking significantly more expensive
  • Output format is not standardised — different implementations use $2a$, $2b$, $2y$ variants

Alternatives

AlgorithmOWASP recommendationKey difference
Argon2idFirst choiceWinner of Password Hashing Competition (2015); memory-hard AND compute-hard; resists GPU & ASIC cracking better than BCrypt
scryptSecond choiceMemory-hard; older than Argon2; configurable N, r, p parameters
PBKDF2Third choice (but FIPS compliant)NIST-approved; required for FIPS 140-2 compliance (government/finance); CPU-only, no memory hardness

16. AutoMapper — Pros, Cons & Alternatives

AutoMapper is a .NET library that automatically maps properties between objects (typically entity → DTO). It uses conventions (matching property names) and user-defined profiles for custom mappings.

In Tally: InvoiceMappingProfile maps Invoice entity → InvoiceDto, CreateInvoiceDto → Invoice, etc. The mapping is configured once in the profile and used throughout the service layer via injected IMapper.

Pros

  • Eliminates boilerplate — no manual property assignment for simple mappings
  • Centralised mapping config — all mappings in one place (the profile); easy to audit
  • Supports complex projections — ProjectTo<T>() translates to SQL SELECT columns directly, avoiding over-fetching
  • Compile-time configuration validation — AssertConfigurationIsValid() catches missing mappings at startup

Cons

  • Magic / implicit behaviour — a renamed property silently stops mapping; hard to debug without config validation
  • Reflection overhead — uses reflection internally; hot paths with thousands of mappings/second may be measurably slower than manual mapping
  • Overkill for simple DTOs — 3-line mapping profile that maps 4 identical properties is often clearer written manually
  • Learning curve — ForMember(), ReverseMap(), ProjectTo(), Ignore() have subtle behaviours
  • Version 12+ breaking changes — AutoMapper v12 removed static mapper; forced DI injection

Alternatives

MapsterCode-gen, reflection, or source-genFaster than AutoMapper; source generator option produces zero-reflection mapping code
Manual mappingExtension methods or constructorsFully explicit, zero magic, zero dependencies; best for small DTOs where the mapping is obvious

17. MailKit

MailKit is a cross-platform .NET email library by Jeffrey Stedfast. It implements IMAP, POP3, and SMTP using MimeKit for message construction. It replaced System.Net.Mail as the recommended .NET email library because SmtpClient was marked obsolete.

Why MailKit over SmtpClient

  • System.Net.Mail.SmtpClient is not disposable correctly, doesn't support modern auth (OAuth2, XOAUTH2), and has poor async support
  • MailKit supports STARTTLS properly, OAuth2 authentication, SMTP AUTH mechanisms (PLAIN, LOGIN, XOAUTH2)
  • Full MIME support via MimeKit — multipart messages, attachments, HTML + text alternatives
  • Works with all major SMTP providers: Gmail, SendGrid, Mailgun, Amazon SES

In Tally

The EmailService uses MailKit to send password reset emails. Configuration comes from appsettings.json (SMTP host, port, credentials). In production, Gmail SMTP is used with app passwords. The service is abstracted behind IEmailService so it can be mocked in tests.

using MailKit.Net.Smtp;

var message = new MimeMessage();

18. Serilog — Pros & Cons

Serilog is a structured logging library for .NET. Unlike traditional loggers that write plain text strings, Serilog captures log events as structured data — property values are preserved as typed fields, not string-concatenated into a message.

Structured logging vs plain text

// Plain text — hard to query
_logger.LogInformation("Invoice " + id + " created by user " + userId);

// Structured — properties are first-class fields
Log.Information("Invoice {InvoiceId} created by {UserId}", id, userId);

In a structured log store (Seq, Elasticsearch, Loki), you can query InvoiceId = 42 directly. In plain text you'd need to regex-parse the message string.

Pros

  • Structured output — JSON logs are machine-readable; easy to index in ELK, Splunk, Datadog, Grafana Loki
  • Sinks — write to console, file, database, Seq, Elasticsearch, CloudWatch — all configurable without code changes
  • Enrichers — automatically attach context: FromLogContext, machine name, thread ID, correlation IDs
  • Minimal level overrides — suppress verbose EF Core SQL logs while keeping application-level Info logs
  • Rolling files — Tally uses daily rolling with 14-day retention

Cons

  • Configuration complexity — sink configuration via appsettings.json is powerful but verbose
  • Destructuring can be surprising — @ prefix for object destructuring vs {Prop} for scalar must be understood
  • File sink is not centralised — in a multi-instance deployment, logs are split across instances; use a centralised sink (CloudWatch, Seq) instead
In Tally: Serilog uses compact JSON formatting (CompactJsonFormatter) to both console and rolling daily files at /app/logs/tally-.log. EF Core and ASP.NET Core log levels are overridden to Warning to reduce noise. The compact JSON format is ready for ingestion by any log aggregator without further parsing.

19. Vue 3 — Pros & Cons

Vue 3 is a progressive JavaScript framework for building UIs. Its core is a reactivity system based on ES6 Proxies. The Composition API (<script setup>) is the modern way to write Vue components.

Pros

  • Composition API — logic is composed via functions (ref, computed, watch) rather than options objects; enables true code reuse via composables
  • Reactivity via Proxy — fine-grained; only the exact components that read a changed ref re-render
  • Single File Components — template, script, and style in one .vue file; great locality
  • TypeScript-first — Volar plugin gives full type inference inside <template> for <script setup lang="ts">
  • Smaller bundle — tree-shakeable; unused features are not included
  • Gentle learning curve — HTML-based template syntax is approachable; Composition API scales to complex apps

Cons

  • Smaller ecosystem than React — fewer third-party component libraries; some niches are underserved
  • Meta-framework choices — React has Next.js as a clear winner; Vue has Nuxt but with less community mass
  • Two APIs (Options + Composition) — existing codebases may mix them; new devs need to learn which to use
  • Template magic — v-model modifiers, scoped slots, dynamic components can be confusing to debug
  • Corporate backing — maintained by Evan You + community (not a large company); lower "enterprise safety" perception
In Tally: All components use <script setup lang="ts"> with Composition API only. Composables (useInvoices.ts, useValidation.ts) extract reusable logic. The reactive stores (Pinia) drive the UI state. Tests use @vue/test-utils to mount components and @pinia/testing to mock stores.

20. JWT, Auth & OAuth

What is a JWT?

A JSON Web Token is a compact, URL-safe string made of three Base64-URL-encoded parts separated by dots: header.payload.signature.

  • Header — algorithm (HS256, RS256) and token type
  • Payload — claims: sub (subject/user ID), exp (expiry), iss (issuer), aud (audience), jti (JWT ID — unique per token), plus custom claims (role)
  • Signature — HMAC-SHA256(base64(header) + "." + base64(payload), secret)

The server validates the signature. No database lookup is needed to verify a JWT — the signature proves it wasn't tampered with and the expiry is encoded in the payload.

Access + Refresh token pattern (Tally)

Login ──────────────────────▶ API │ └──▶ access token (25 min, JWT) └──▶ refresh token (7 days, opaque, stored in DB) Authenticated request: client ──[Authorization: Bearer <access_token>]──▶ API API validates signature + expiry + Redis blacklist check Silent refresh (1 min before access token expires): client ──[POST /auth/refresh, body: refreshToken]──▶ API API checks RefreshTokens table, issues new token pair, invalidates old refresh token Logout: API blacklists the JTI in Redis (TTL = remaining lifetime) API marks the refresh token as revoked in DB

HS256 vs RS256

HS256 (symmetric)RS256 (asymmetric)
KeyOne shared secret — both issuer and verifier use the same keyPrivate key signs; public key verifies — verifiers don't need the private key
Use whenSingle service issues and verifies tokens (Tally's case)Multiple services need to verify; external parties (OAuth resource servers)
Security riskIf any verifier is compromised, they could forge tokensLeaking the public key is safe; only leaking the private key is dangerous

OAuth 2.0 vs JWT

OAuth 2.0 is an authorisation framework — it defines flows (Authorization Code, Client Credentials, Implicit, Device) for delegating access. JWT is a token format — it can be used as the token inside OAuth flows (as an access token). They are not alternatives to each other.

OpenID Connect (OIDC) is an identity layer on top of OAuth 2.0 that adds a standard id_token (always a JWT) containing user identity claims. OIDC is how "Sign in with Google" works — Google issues an OIDC id_token after OAuth consent.

Known issue in Tally: Tokens are stored in localStorage — readable by any JavaScript on the page (XSS risk). The planned migration is to HttpOnly cookies, which are inaccessible to JavaScript. This is a real trade-off: localStorage is simpler for SPAs (no CSRF concern) but HttpOnly cookies are more secure against XSS.

21. Background DB Jobs

Background jobs are tasks that run outside the HTTP request-response cycle — either on a schedule, triggered by an event, or continuously.

In Tally: OverdueInvoiceJob

This is an IHostedService — a .NET interface for long-running background tasks. The CLR starts it on app startup and calls StopAsync() on shutdown. The job:

  1. Calculates the time until next midnight UTC
  2. Awaits that delay (the thread is released; no CPU consumption while waiting)
  3. Issues a bulk ExecuteUpdateAsync() to mark all Sent invoices past their due date as Overdue
  4. Loops back to step 1
// EF Core 7+ bulk update — single SQL UPDATE, no change tracking
await _context.Invoices
    .Where(i => i.Status == InvoiceStatus.Sent && i.DueDate < DateTime.UtcNow)
    .ExecuteUpdateAsync(s => s.SetProperty(i => i.Status, InvoiceStatus.Overdue));

22. AppDbContext & How EF Core Handles It

AppDbContext is the unit-of-work and gateway to the database in EF Core. It inherits from DbContext and exposes DbSet<T> properties for each entity.

Registration and lifetime

In ASP.NET Core, AppDbContext is registered with AddDbContext<AppDbContext>(). This registers it as scoped by default — one instance per HTTP request. This is intentional: a scoped DbContext means all repositories and services within a single request share the same context (and thus the same change tracker and the same transaction if needed).

builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseNpgsql(connectionString));
Never register DbContext as Singleton. DbContext is not thread-safe — it holds state (change tracker, open connections). A singleton DbContext shared across requests causes concurrency exceptions and stale data.

Change tracking

EF Core tracks every entity loaded from the database. When you call SaveChangesAsync(), it compares current state to the original snapshot and issues the minimal set of SQL statements (INSERT, UPDATE, DELETE).

  • EntityState.Added → INSERT
  • EntityState.Modified → UPDATE (only changed columns)
  • EntityState.Deleted → DELETE
  • EntityState.Unchanged → no SQL

AsNoTracking for reads

When you only need to read data (no update), call .AsNoTracking() on the query. EF Core skips change tracking — no snapshot is stored in memory, reducing allocation and improving performance.

// Read-only query — no need to track
var invoices = await _context.Invoices
    .AsNoTracking()
    .Where(i => i.UserId == userId)
    .ToListAsync();

Connection pooling

EF Core does not hold a physical database connection open for the lifetime of the DbContext. It uses ADO.NET connection pooling (Npgsql in Tally's case). A physical connection is only acquired when a query is actually executed, and released back to the pool immediately after. This allows a small pool of physical connections to serve thousands of concurrent HTTP requests.

Migrations in Tally

Migrations run automatically on startup via Database.Migrate() in Program.cs. EF Core checks the __EFMigrationsHistory table and applies any pending migrations. This is acceptable for a single-instance app but risky for multi-instance deployments (race condition on startup) — use Database.MigrateAsync() with a distributed lock or a separate migration job in that case.

23. IaC & IaaS

IaaS — Infrastructure as a Service

IaaS is a cloud service model where the provider (AWS, Azure, GCP) supplies raw compute, storage, and networking infrastructure. You provision virtual machines, storage volumes, and network configuration. You manage the OS, runtime, and application. The provider manages the physical hardware.

  • Examples: AWS EC2, Azure Virtual Machines, GCP Compute Engine
  • Tally runs on AWS EC2 — an IaaS virtual machine. You control the OS (Amazon Linux), Docker installation, and deployment scripts.

Contrast with PaaS and SaaS

ModelYou manageProvider managesExample
IaaSOS, runtime, app, dataHardware, networking, virtualisationEC2, Azure VM
PaaSApp, dataOS, runtime, scaling, patchingHeroku, AWS Elastic Beanstalk, Azure App Service
SaaSConfiguration onlyEverythingGitHub, Slack, Salesforce
CaaSContainer images, configContainer orchestration, OSAWS ECS, Azure Container Apps, GKE

IaC — Infrastructure as Code

IaC means declaring your infrastructure in code (version-controlled, reviewed, repeatable) rather than clicking through a web console. Key properties:

  • Idempotent — running the same IaC twice produces the same result, no duplicates
  • Version controlled — infrastructure changes are tracked in git with diffs and history
  • Reproducible — spin up an identical environment in a new region with one command
  • Self-documenting — the code describes exactly what exists

Tally uses Terraform as its IaC tool. The terraform/ec2/main.tf provisions the EC2 instance, Elastic IP, security groups, IAM role, and Secrets Manager secret. The terraform/main.tf manages the PostgreSQL database schema and seed data.

24. Containers — Overview

A container is a lightweight, isolated unit of software that packages an application and all its dependencies (runtime, libraries, config) into a single image. Containers share the host OS kernel but are isolated via Linux namespaces (process, network, mount, user) and cgroups (resource limits).

Containers vs VMs

Virtual MachineContainer
IsolationFull OS virtualisation (hypervisor)Kernel namespaces + cgroups
Startup timeMinutes (full OS boot)Milliseconds (process start)
SizeGBs (includes full OS)MBs (just app + dependencies)
Resource overheadHigh (each VM has its own kernel)Low (shared kernel)
Security boundaryStronger (hypervisor separation)Weaker (shared kernel — a kernel exploit can escape)
PortabilityHeavy (VM image tied to hypervisor)Runs anywhere Docker/OCI runtime is installed

OCI and Docker

The Open Container Initiative (OCI) defines the container image format and runtime spec. Docker popularised containers and its image format became the OCI standard. Alternatives to Docker Desktop: Podman (daemonless, rootless), nerdctl (containerd CLI), Lima (macOS VM for Linux containers).

25. Docker

Docker is the toolchain for building, distributing, and running container images. Key concepts: image (immutable blueprint, built from a Dockerfile), container (running instance of an image), registry (Docker Hub, ECR, GHCR — image storage), volume (persistent storage mounted into a container), network (virtual networks between containers).

Tally's Docker setup

Dockerfile → tally_api image (ASP.NET Core 9 on mcr.microsoft.com/dotnet/aspnet:9.0) Dockerfile.frontend → tally_frontend image (nginx:alpine serving compiled Vue) docker-compose.yml: ┌─────────────────────────────────────────────────────┐ │ tally_frontend (nginx) — exposes :80, :443 │ │ │ │ │ ├── serves /app/dist (Vue SPA) │ │ └── proxies /api/ ──▶ tally_api:8080 │ │ │ │ tally_api (.NET 9) — internal :8080 │ │ ├── depends on: postgres (healthy) │ │ └── depends on: redis (healthy) │ │ │ │ tally_db (postgres:16) — internal :5432 │ │ tally_redis (redis:7) — internal :6379 │ └─────────────────────────────────────────────────────┘ Ports 5432 and 6379 NOT exposed to host — containers reach each other via Docker's internal bridge network.

Key Docker concepts used in Tally

  • Multi-stage build — the Dockerfile uses a build stage (mcr.microsoft.com/dotnet/sdk:9.0) to compile the app, then copies only the output to a smaller runtime image (aspnet:9.0). The SDK (hundreds of MBs) is not in the final image.
  • Health checks — Postgres and Redis containers define healthcheck in docker-compose.yml; the API container starts only when they are healthy (depends_on: condition: service_healthy).
  • Named volumes — postgres_data and redis_data persist data across container restarts
  • Environment variable injection — secrets (JWT_KEY, POSTGRES_PASSWORD) come from the host environment, never baked into the image
Why Postgres and Redis ports are not exposed to the host: Exposing port 5432 to the public internet would allow anyone to attempt a database connection. By keeping them on Docker's internal network, only containers on the same Compose network can reach them.

26. Terraform & How It's Used in Tally

Terraform (by HashiCorp) is a declarative IaC tool. You describe the desired state of infrastructure in HCL (HashiCorp Configuration Language); Terraform computes a plan (diff) and applies it. It tracks what it has created in a state file.

Core concepts

  • Provider — plugin that talks to an API (AWS, PostgreSQL, GitHub). Tally uses the cyrilgdn/postgresql provider and the aws provider in the ec2 module.
  • Resource — a managed piece of infrastructure (postgresql_database, aws_instance)
  • Variable — input parameter to a module; passed at runtime
  • Output — value exported from a module (e.g. the connection string)
  • State — Terraform tracks created resources in terraform.tfstate. In production, state is stored remotely (S3 + DynamoDB lock) to allow team collaboration and prevent conflicts.
  • Plan — terraform plan shows what will be created/changed/destroyed without doing it
  • Apply — terraform apply makes the changes; idempotent (re-running has no effect if state matches desired)

What Tally's Terraform does

terraform/main.tf — manages the PostgreSQL database schema (using the postgresql provider): creates the database, generates a seed.sql file as a local_file resource, and applies it via a null_resource + local-exec provisioner (runs psql on the local machine).

terraform/ec2/main.tf — provisions AWS infrastructure:

  • EC2 instance (Amazon Linux, t2.micro/t3.micro)
  • Elastic IP — static public IP so the DNS record doesn't need updating on stop/start
  • Security group — allows 80 (HTTP), 443 (HTTPS), 22 (SSH)
  • IAM instance role — grants the EC2 instance permission to read from Secrets Manager
  • Secrets Manager secret — stores the JWT key, Postgres password, email credentials
  • EC2 user-data script — installs Docker on first boot, writes a fetch-secrets.sh script that pulls secrets from Secrets Manager at deploy time

27. Vite & How It's Used in Tally

Vite (French for "fast") is a next-generation frontend build tool by Evan You (Vue's creator). It serves as both a dev server and a production bundler (using Rollup under the hood for prod).

Why Vite is fast in development

Traditional bundlers (Webpack) bundle the entire application before serving. Vite uses native ES Modules — the browser imports files directly. The dev server only transforms a file when the browser requests it (lazy transformation). Hot Module Replacement is near-instant because only the changed module is re-evaluated, not the whole bundle.

Production build

In production, npm run build runs vue-tsc (TypeScript type checking) then vite build. Vite uses Rollup to bundle and tree-shake the output. Code splitting is automatic — each route chunk is a separate file. The output is hashed filenames (main.a3b4c5d6.js) for cache busting.

In Tally

  • Dev server: npm run dev — Vite serves modules natively, connects to the API at VITE_API_URL from .env.development
  • Production: vite build compiles the Vue app to static files in dist/; the Dockerfile.frontend copies this dist/ into an nginx container
  • @vitejs/plugin-vue — transforms .vue SFCs
  • VITE_API_URL — in production is set to / because nginx proxies /api/ on the same origin — no cross-origin requests, no CORS

28. Pinia & How It's Used in Tally

Pinia is the official Vue 3 state management library. It replaced Vuex. Each store is a composable — you define state, getters (computed), and actions (async functions) in a single defineStore() call.

Why Pinia over Vuex

  • No mutations — Vuex required commit('MUTATION_NAME'); Pinia lets you modify state directly in actions
  • TypeScript-first — full type inference for state, getters, and actions without boilerplate
  • DevTools integration — Vue DevTools shows store state, actions, and time-travel debugging
  • Simpler API — defineStore('auth', () => { ... }) looks like a composable

Tally's stores

StoreState heldKey actions
authStoreJWT token, refresh token, user, expirylogin(), logout(), register(), silentRefresh(), restoreSession()

Silent refresh in authStore: On login, authStore parses the JWT expiry from the token payload and calls setTimeout to schedule silentRefresh() 60 seconds before expiry. On page reload, restoreSession() reads tokens from localStorage, checks if the access token is still valid, and either re-arms the timer or immediately refreshes if already expired. This makes the session feel seamless — the user is never unexpectedly logged out mid-session.

29. Secrets Manager & How Secrets Are Managed in Tally

The problem

Secrets (database passwords, JWT signing keys, SMTP credentials) must never be committed to source control, embedded in Docker images, or stored in Terraform state in plaintext.

In development (local)

ASP.NET Core's User Secrets (dotnet user-secrets) stores development secrets in a JSON file outside the project directory (~/.microsoft/usersecrets/<user-secrets-id>/secrets.json). The UserSecretsId in InvoiceTrackerAPI2.csproj ties this project to its secrets file. They are never committed. They are loaded automatically by ASP.NET Core in Development environment only.

In production (AWS EC2)

Tally uses AWS Secrets Manager:

  1. Terraform creates a Secrets Manager secret and an IAM role that grants the EC2 instance secretsmanager:GetSecretValue permission
  2. The EC2 instance has the IAM role attached — no AWS access keys are stored anywhere on the instance
  3. A fetch-secrets.sh script (written by Terraform's user-data) runs at deploy time, calls aws secretsmanager get-secret-value, and writes the values to a .env file on the EC2 host
  4. Docker Compose reads from the .env file when starting containers — secrets are injected as environment variables
Secrets Manager (AWS) │ └── IAM role attached to EC2 (no static access keys) │ └── fetch-secrets.sh pulls secrets at deploy time │ └── writes to /home/ec2-user/tally/.env │ └── Docker Compose injects as env vars │ └── ASP.NET Core reads via IConfiguration

Why this approach

  • No secrets in git — the repo contains no real credentials anywhere
  • No secrets in the Docker image — secrets are injected at runtime, not baked into the image layer
  • No secrets in Terraform state — the Secrets Manager secret resource is in state, but the secret value is set manually via AWS console or CLI, not via Terraform
  • Rotation support — Secrets Manager supports automatic rotation; rotating a secret doesn't require a code or infrastructure change
  • Audit trail — AWS CloudTrail logs every GetSecretValue call
The .env file on disk is a risk: Anyone with SSH access to the EC2 instance can read it. This is acceptable for a single-instance project; in a higher-security context, you'd inject secrets directly from Secrets Manager into the container at runtime (using ECS task definitions with secrets: blocks) rather than writing to a file on disk.
